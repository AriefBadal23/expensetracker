# 💸 ExpenseFlow

A full-stack expense tracking application designed with a strong focus on backend architecture and secure data handling.

The system consists of a React frontend and an ASP.NET Core backend API, where transactions are organized into categories ("buckets") and summarized into monthly insights.

---

## 🛠️ Technologies

- React
- TypeScript
- ASP.NET Core
- .NET
- PostgreSQL
---

## 🚀 Features

- Manage personal income and expenses using a bucket-based system.
- Categorize transactions and generate monthly summaries.
- Structured backend with clear separation between controllers, services, and data access.
- Input validation and preparation for secure authentication and authorization.
- Designed for maintainability and extensibility.

---

## 🧠 Key Technical Decisions

- Implemented a service layer to separate business logic from controllers.
- Used dependency injection to keep components loosely coupled.
- Designed the backend with scalability and maintainability in mind.
- Applied secure coding practices such as input validation and safe data handling.
- Structured API responses to support a clean frontend integration.

---

## 📍 The Process

The project started with designing the backend API using ASP.NET Core, focusing on a clear and maintainable architecture. The initial implementation covered core functionality for managing transactions, including creating, updating, and retrieving expense data.

A service layer was introduced early to separate business logic from controllers, making the system easier to extend and reason about.

On the frontend, React components were built to interact with the API and present transaction data. State management was used to keep the UI in sync with backend changes.

As the project evolved, the focus shifted towards improving code quality and applying secure coding practices learned during my cybersecurity minor. This included implementing input validation, handling edge cases, and preparing the system for authentication and authorization.

The goal was not just to build a working application, but to structure it in a way that reflects real-world backend systems.
